\section{Deployment and Runtime Configuration}\label{sec:deployment}

The browser-based studio runs in two environments: a \textbf{local Docker Compose stack} for development and thesis
demonstrations, and a \textbf{production stack} on Railway with Cloudflare DNS and TLS in front of
\texttt{motivo-studio.eu}.
Both share the same logical shape (browser to Next.js client, same-origin \texttt{/api/*} proxy to Express, private
database networking, embedded \texttt{motivoc} in the server image) but differ in TLS termination and gateway
components.

\subsection{Local Docker Compose stack}\label{subsec:local-docker}

Docker Compose wires four cooperating services plus a named Postgres volume: PostgreSQL, a multi-stage server image
that installs \texttt{motivoc} and runs migrations before listen, a Next.js client on the Compose network, and an nginx
gateway that redirects HTTP to \texttt{https://motivo-studio.local}, terminates HTTPS with a local self-signed
certificate, proxies \texttt{/api/*} to Express (stripping the prefix), and forwards other paths to the client.

Docker pins the compiler toolchain, Node runtimes, and service wiring so thesis results and demos do not depend on ad
hoc host installations.

\subsection{Production: Railway, Cloudflare, and domain}\label{subsec:production-railway}

Production serves \texttt{https://motivo-studio.eu} and \texttt{https://www.motivo-studio.eu}.

\topic{Railway.} Three services in the Amsterdam region: a public Next.js client image, a private Express server image
with embedded \texttt{motivoc} (reachable from the client via internal networking), and managed PostgreSQL on private
networking.

Pushes to \texttt{main} trigger module-scoped CI first; Railway deploys only after the relevant pipeline succeeds,
with path-scoped watches so client-only changes redeploy the client and server or compiler changes redeploy the
backend. Migrations run before the server accepts traffic, matching the local container entrypoint.

\topic{Cloudflare.} Fronts \texttt{motivo-studio.eu}: apex and \texttt{www} DNS proxy to the Railway origin with
\textbf{Full (Strict)} TLS. Browsers use the same origin for pages and \texttt{/api/*}, matching the Next.js rewrite
design locally.

\subsection{CI coupling to deploy}\label{subsec:ci-deploy-gate}

Module CI workflows (compiler, server, client, and this paper) run on GitHub Actions for each push. Railway treats a
green \texttt{main} pipeline as a deploy gate before promoting new container images, so failing tests block production
rollout without manual override.
